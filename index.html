The Problem

Incident knowledge is trapped and too sensitive to share.

Failure Silos

Large enterprises only experience their own technical failures. Even if they have the capital to build an internal tool, they miss critical edge cases seen elsewhere across the industry because competitors will never safely connect to a rival's proprietary platform.

Postmortems are too sensitive for cloud LLMs

Incident reports contain internal hostnames, software versions, past vulnerabilities, and sometimes credentials. Sending these to GPT-4 or any external API means your cloud provider can read them.

Time-to-resolution is the metric that matters

Every minute of an outage costs revenue and trust. The fastest resolution path is a past incident with a matching root cause — but finding it in a folder of hundreds of documents is too slow.

Coldstart Friction

New companies don't have a big log history to refer to, making it difficult to recognize recurring failure modes and troubleshoot early infrastructure anomalies effectively.

Query your history. Get cited answers. Trust the hardware.

AIIRA runs the full pipeline — embedding, retrieval, and LLM inference — inside a hardware-sealed Confidential VM. The cloud provider cannot read it. We cannot read it. Only authorised users can.

Clients receive a cryptographic attestation proof before uploading a single document. Trust is established by hardware, not policy.

  • Semantic search surfaces matches that keyword search misses
  • Answers cite source files — engineers can verify every claim
  • No external API calls — LLM inference runs locally inside the CVM
  • Intel TDX memory encryption: data stays sealed during processing
  • Remote attestation via dstack SDK — verifiable before any data moves

Architecture

Everything inside the enclave. Nothing outside it.

Client
On-call engineer submits a natural-language query over HTTPS. Before uploading any documents, client verifies the cryptographic attestation proof per session to confirm exact code is running.
↓ TLS (Dstack Gateway) ↓
TEE BOUNDARY — Intel TDX CVM
FastAPI
REST endpoint inside the CVM. Receives query, orchestrates the pipeline, returns grounded answer with source citations. All processing sealed inside hardware-encrypted memory.
TEE BOUNDARY — Intel TDX CVM
Embedding
sentence-transformers (all-MiniLM-L6-v2) converts the query to a vector embedding. Runs locally — no external embedding API call. Vector stays inside the enclave.
TEE BOUNDARY — Intel TDX CVM
FAISS Index
Sealed in-memory vector index over all ingested postmortems. Approximate nearest-neighbour search returns top-k semantically relevant chunks. Index never leaves the CVM.
TEE BOUNDARY — Intel TDX CVM
LLM (Ollama)
Local 8B parameter model synthesises a grounded answer from retrieved chunks. No external API call. Answer cites source documents. Inference hardware-isolated from operator and cloud provider.

Attestation
On demand, the CVM generates a TDX Quote — a signed hardware report containing the MRENCLAVE measurement of the running code. Verifiable independently at proof.t16z.com before any data is uploaded.

Security Model - CIA

Three guarantees. All hardware-enforced.

01

Confidentiality

Intel TDX encrypts CVM memory at the hardware level. The cloud provider, the hypervisor, and the OS cannot read postmortems or queries being processed. Data stays sealed during computation — not just at rest.

02

Integrity

TDX measures the CVM's initial memory state. Any modification to the running code changes the measurement. Tampering is detectable before any client uploads data — by anyone, independently.

03

Attestation

A TDX Quote is a signed hardware report proving which exact code is running. Signed by Intel's attestation key — not ours. Clients verify this independently before trusting us with a single document. Trust is cryptographic, not contractual.
