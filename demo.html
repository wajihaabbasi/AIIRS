The Problem

Incident knowledge is trapped and too sensitive to share.

🔍

Keyword search fails under pressure

At 2am, engineers describe symptoms — not root causes. A keyword search for "API latency" misses the postmortem titled "database connection pool exhaustion". Semantic search doesn't.

🔒

Postmortems are too sensitive for cloud LLMs

Incident reports contain internal hostnames, software versions, past vulnerabilities, and sometimes credentials. Sending these to any external API means your cloud provider can read them.

⏱️

Time-to-resolution is the metric that matters

Every minute of an outage costs revenue and trust. The fastest resolution path is a past incident with a matching root cause — but finding it across hundreds of documents is too slow.

🏗️

Small teams can't build this in-house

Building a confidential AI pipeline requires deep expertise in hardware security, vector databases, and MLOps. Most companies don't have that — and can't afford to hire it.

Architecture

Everything inside the enclave. Nothing outside it.

👤Client
On-call engineer submits a natural-language query over HTTPS. Before uploading any documents, the client verifies the cryptographic attestation proof to confirm the exact code is running.
↓ TLS (Dstack Gateway) ↓
TEE BOUNDARY — Intel TDX CVM
⚡FastAPI
REST endpoint inside the CVM. Receives query, orchestrates the pipeline, returns grounded answer with source citations. All processing sealed inside hardware-encrypted memory.
TEE BOUNDARY — Intel TDX CVM
🔢Embedding
sentence-transformers (all-MiniLM-L6-v2) converts the query to a vector embedding. Runs locally — no external embedding API call. The vector stays inside the enclave.
TEE BOUNDARY — Intel TDX CVM
📚FAISS Index
Sealed in-memory vector index over all ingested postmortems. Approximate nearest-neighbour search returns top-k semantically relevant chunks. Index never leaves the CVM.
TEE BOUNDARY — Intel TDX CVM
🧠LLM (Ollama)
Local 8B parameter model synthesises a grounded answer from retrieved chunks. No external API call. Answer cites source documents. Inference is hardware-isolated from operator and cloud provider.
↓ dstack SDK · get_quote() ↓
🔐Attestation
On demand, the CVM generates a TDX Quote — a signed hardware report containing the MRENCLAVE measurement of the running code. Verifiable independently at proof.t16z.com before any data is uploaded.

Security Model

Three guarantees. All hardware-enforced.

01

Confidentiality

Intel TDX encrypts CVM memory at the hardware level. The cloud provider, the hypervisor, and the OS cannot read postmortems or queries being processed. Data stays sealed during computation — not just at rest.

02

Integrity

TDX measures the CVM's initial memory state. Any modification to the running code changes the measurement. Tampering is detectable before any client uploads data — by anyone, independently.

03

Attestation

A TDX Quote is a signed hardware report proving which exact code is running. Signed by Intel's attestation key — not ours. Clients verify this independently before trusting us with a single document.
